# Project Structure

## Directory Tree

```
rag-product-recommend/
├── pyproject.toml              # Dependencies & project metadata
├── uv.lock                     # Lockfile (like package-lock.json)
├── .env                        # API keys (not committed)
├── CLAUDE.md                   # AI context & coding rules
│
├── src/                        # Core business logic
│   ├── ingestion/              # Data loading & normalization
│   ├── embedding/              # Embedding & Vector DB
│   ├── retrieval/              # Product retrieval & search
│   ├── generation/             # LLM generation & prompts
│   ├── pipeline/               # Orchestration layer
│   │   ├── recommend/          #   Recommendation domain logic
│   │   └── compare/            #   Comparison domain logic
│   └── utils/                  # Shared utilities
│
├── api/                        # FastAPI layer
│   ├── routes/                 #   API route handlers
│   └── middleware/             #   Request/response middleware
│
├── tests/                      # Test suite
│   ├── unit/                   #   Unit tests
│   └── integration/            #   Integration tests
│
├── evaluation/                 # RAG quality evaluation

├── docs/                       # MkDocs documentation source
```

---

## Root Files

| File | Purpose | When to update |
| ---- | ------- | -------------- |
| `pyproject.toml` | Project metadata, dependencies, build config | When adding/removing a dependency (`uv add`), changing Python version, or updating project metadata |
| `uv.lock` | Pinned dependency versions (auto-generated) | Never edit manually — regenerated by `uv lock` or `uv add` |
| `.env` | Environment variables (API keys, log level) | When adding a new provider or changing environment config. **Not committed to git.** |
| `.env.example` | Template for `.env` | When a new environment variable is required — add it here so other devs know |
| `CLAUDE.md` | AI coding rules + project structure reference | When project structure changes, new conventions are adopted, or language rules need updating |
| `mkdocs.yml` | MkDocs Material site configuration | When adding new doc pages, changing navigation, or updating theme/plugins |
| `.gitignore` | Git ignore rules | When new generated/secret files need to be excluded |

## `src/` — Core Business Logic

All domain logic lives here. This is a pure Python package with no web framework dependency — it can be used independently of FastAPI.

### `src/ingestion/` — Data Loading & Normalization

**Purpose:** Load raw product data from various sources, clean it, parse specifications, and chunk it into units suitable for embedding.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `product_loader.py` | Load product data from JSON/CSV files | When supporting a new data source format (e.g., XML, database export) |
| `review_loader.py` | Load user reviews and ratings | When adding a new review source or changing review schema |
| `data_cleaner.py` | Normalize text, fix encoding, remove duplicates | When adding new cleaning rules (e.g., currency normalization, HTML stripping) |
| `spec_parser.py` | Parse product specifications into structured key-value pairs | When supporting a new product category with different spec formats |
| `chunker.py` | Field-based chunking — split product data into embeddable chunks | When changing chunking strategy (e.g., chunk size, overlap, field grouping) |
| `price_tracker.py` | Track price history and detect price changes | When adding price alert logic or new price data sources |

**When to add a new file:** When you need a new data source type (e.g., `api_loader.py` for a REST API) or a new preprocessing step (e.g., `deduplicator.py`).

### `src/embedding/` — Embedding & Vector DB

**Purpose:** Convert text into vectors and manage the vector database (ChromaDB / Qdrant).

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `product_embedder.py` | Embed text into vectors using OpenAI `text-embedding-3-small` | When switching embedding model or adding a new embedding provider |
| `multi_field_embedder.py` | Generate separate embeddings per product field (name, description, specs) | When changing which fields get their own embeddings or adjusting field weights |
| `vector_store.py` | CRUD operations for ChromaDB/Qdrant — create collection, upsert, query, delete | When switching vector DB provider, adding new query patterns, or changing similarity metric |

**When to add a new file:** When adding a new embedding strategy (e.g., `sparse_embedder.py` for BM25 vectors) or a new vector DB backend.

### `src/retrieval/` — Product Retrieval & Search

**Purpose:** Given a user query, retrieve the most relevant products from the vector store using a combination of semantic search, keyword matching, and metadata filtering.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `product_retriever.py` | Main retriever — combines filter extraction, embedding, vector query, and scoring into one flow | When changing the retrieval pipeline order or adding a new retrieval step |
| `hybrid_search.py` | Combine semantic search + keyword search (BM25) + metadata filters | When implementing BM25, adding a new search strategy, or changing fusion weights |
| `filter_engine.py` | Extract structured filters (price, brand, category, rating) from Vietnamese natural language queries | When supporting a new filter type (e.g., color, storage size) or adding new keyword patterns |
| `similarity_scorer.py` | Compute composite relevance score from semantic similarity + metadata signals | When changing scoring weights or adding new scoring dimensions |
| `reranker.py` | Cross-encoder reranking using `ms-marco-MiniLM-L-6-v2` for more accurate relevance scoring | When switching reranker model or adding reranking strategies (e.g., LLM-based reranking) |

**When to add a new file:** When adding a new search strategy (e.g., `bm25_search.py`), a new filter type that's complex enough to warrant its own module, or a new scoring algorithm.

### `src/generation/` — LLM Generation

**Purpose:** Call LLM providers, manage prompt templates, parse responses, and validate input/output.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `llm_client.py` | Unified client for Anthropic, OpenAI, and Gemini — all LLM calls go through here | When adding a new LLM provider or changing the client interface |
| `response_parser.py` | Parse structured JSON from LLM text output | When changing the expected response format or adding a new response type |
| `guardrails.py` | Input validation (query length, injection detection) and output validation (well-formed JSON, no hallucinations) | When adding new safety checks or adjusting validation thresholds |

**`src/generation/prompt_templates/`** — One file per use case, each exporting `SYSTEM_PROMPT` and `USER_PROMPT_TEMPLATE` as module-level constants.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `recommend_prompt.py` | Prompt for product recommendation responses | When tuning recommendation output quality or changing response format |
| `compare_prompt.py` | Prompt for product comparison responses | When adjusting comparison criteria or output structure |
| `review_summary_prompt.py` | Prompt for summarizing user reviews | When adding review summarization features |

**When to add a new file:** When creating a new pipeline type (e.g., `faq_prompt.py` for a FAQ pipeline) or a new LLM provider that needs its own client module.

### `src/pipeline/` — Orchestration Layer

**Purpose:** Wire together retrieval, scoring, and generation into end-to-end pipelines. This is the "glue" layer.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `rag_router.py` | Classify incoming queries (RECOMMEND / COMPARE / INFO / HYBRID) using regex patterns | When adding a new query type or improving classification accuracy |
| `config.py` | `PipelineConfig` dataclass — loaded from `configs/settings.yaml` | When adding a new config parameter (e.g., new provider, new threshold) |
| `recommend_pipeline.py` | End-to-end recommendation flow: intent → retrieve → score → LLM → response | When changing the recommendation pipeline steps or adding pre/post-processing |
| `compare_pipeline.py` | End-to-end comparison flow: get products → compare → LLM → response | When changing the comparison pipeline steps |

**`src/pipeline/recommend/`** — Recommendation domain logic (called by `recommend_pipeline.py`).

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `engine.py` | Main recommendation engine — orchestrates intent parsing, retrieval, and scoring | When changing the recommendation algorithm |
| `user_intent_parser.py` | Parse Vietnamese queries into structured intent (budget, use_case, priorities, brand) | When supporting new intent fields or improving NLP accuracy |
| `scoring.py` | Multi-criteria product scoring (price match, rating, feature overlap) | When adding new scoring criteria or adjusting weights |
| `personalization.py` | Boost scores based on user history and preferences | When adding personalization features (e.g., collaborative filtering) |

**`src/pipeline/compare/`** — Comparison domain logic (called by `compare_pipeline.py`).

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `comparator.py` | Compare N products across all specification dimensions | When changing comparison logic or supporting new product types |
| `spec_aligner.py` | Normalize and align specs across products to a common schema | When handling new spec formats or improving alignment accuracy |
| `formatter.py` | Format comparison results as Markdown tables | When changing output format (e.g., HTML, structured JSON) |
| `pros_cons_extractor.py` | Extract advantages and disadvantages for each product | When improving pros/cons extraction logic |

**When to add a new file in `src/pipeline/`:** When creating an entirely new pipeline (e.g., `faq_pipeline.py` + `faq/` subfolder for a FAQ answering pipeline).

### `src/utils/` — Shared Utilities

**Purpose:** Cross-cutting helpers used by multiple modules.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `logger.py` | Structured logging setup | When changing log format or adding log destinations |
| `cache.py` | Caching layer (Redis-backed) for embeddings and LLM responses | When adding new cache strategies or changing TTL |
| `helpers.py` | General-purpose utility functions | When adding a small helper that doesn't belong in a specific module |

**When to add a new file:** When a utility grows beyond a few functions and deserves its own module (e.g., `metrics.py` for performance tracking).

## `api/` — FastAPI Layer

**Purpose:** HTTP interface to the pipeline. Thin layer — routes validate input, call pipeline factories from `deps.py`, and return JSON.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `app.py` | FastAPI application entry point — mounts routes, middleware, CORS, health check | When adding new route groups or middleware |
| `schemas.py` | Pydantic request/response models | When changing API contract (new fields, new endpoints) |
| `deps.py` | Dependency injection factory functions (`get_retriever()`, `get_llm_client()`, etc.) | When adding a new pipeline component or changing how components are wired |

**`api/routes/`** — One file per API domain.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `recommend.py` | `POST /api/recommend` handler | When changing recommendation request/response format |
| `compare.py` | `POST /api/compare` handler | When changing comparison request/response format |
| `search.py` | `POST /api/search` handler | When adding search features or filters |

**`api/middleware/`** — Request/response middleware.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `rate_limit.py` | Rate limiting per IP/API key | When adjusting rate limits or adding per-user limits |
| `error_handler.py` | Global exception handling — converts exceptions to consistent JSON error responses | When adding new exception types or changing error format |

**When to add a new file:** When creating a new API domain (e.g., `routes/admin.py`) or a new middleware (e.g., `middleware/auth.py` for authentication).

## `tests/` — Test Suite

**Purpose:** Automated tests using pytest. Mirrors the `src/` structure.

| Path | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `conftest.py` | Shared fixtures (mock configs, sample data, test clients) | When adding fixtures needed by multiple test files |
| `unit/` | Unit tests — test individual classes/functions in isolation with mocks | Add a test file for every new module in `src/` or `api/` |
| `integration/` | Integration tests — test full pipeline flows with real (or docker) dependencies | Add tests when a new pipeline or API endpoint is created |

**Naming convention:** `test_<module_name>.py` (e.g., `test_filter_engine.py` for `src/retrieval/filter_engine.py`).

## `evaluation/` — RAG Quality Evaluation

**Purpose:** Measure retrieval and generation quality. Run offline to benchmark changes.

| File | Purpose | When to add/update |
| ---- | ------- | ------------------- |
| `eval_recommend.py` | Evaluate recommendation pipeline (retrieval recall, ranking accuracy, response quality) | When changing retrieval or scoring logic — run before/after to compare |
| `eval_compare.py` | Evaluate comparison pipeline (spec alignment accuracy, analysis quality) | When changing comparison logic |
| `test_cases.json` | Ground-truth test cases with expected results | When adding new product categories or edge cases |

**When to add a new file:** When creating a new evaluation metric (e.g., `eval_latency.py`) or a new pipeline to evaluate.

## Key Conventions

| Convention | Rule |
| ---------- | ---- |
| Imports | Always absolute from project root: `from src.retrieval.filter_engine import FilterEngine` |
| Config | `PipelineConfig` dataclass loaded from `configs/settings.yaml` |
| LLM calls | Always through `src/generation/llm_client.py` — never call provider APIs directly |
| Vector DB | Always through `src/embedding/vector_store.py` |
| Prompt templates | Module-level constants: `SYSTEM_PROMPT`, `USER_PROMPT_TEMPLATE` |
| API dependencies | Factory functions in `api/deps.py` (e.g., `get_retriever()`, `get_llm_client()`) |
| User-facing text | Vietnamese |
| Code & comments | English |
| Package management | `uv` only — never `pip install` |
| New modules | Always add a corresponding test file in `tests/unit/` |
